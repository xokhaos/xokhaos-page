A living system map / capability overview

MOONLASH

3D AI Vtuber Red Head connected to a complete 3D environment and a persistent 24/7 Discord presence. It controls conversation, memory, voice, vision, characters, cameras, lights, animation, tracking, clothing, props, particles, games, media, scheduling, and recovery.

Live inputs / event normalization

Multichannel event ingestion.

Messages, voice, environmental sounds, visual posts, community activity, and timers are normalized before scheduling.

Voice in

Speech recognition and turn detection.

Pre-roll, onset, silence, endpointing, interim text, final text, noise rejection, and reconnects protect live conversation.

Vision in

Media download and visual analysis.

Images, animation, video, and shared media can be interpreted, summarized, and placed into context.

Microphone input

Live microphone acquisition.

Audio devices are selected, sampled continuously, buffered before speech onset, and released cleanly during shutdown or recovery.

Speech boundaries

Voice-activity detection.

Multiple detection paths identify speech onset, preserve pre-speech audio, measure trailing silence, and decide when an utterance is complete.

Speech-to-text

Streaming partial and final transcripts.

Realtime recognition produces interim text for responsiveness and committed text for context, with turn resumes, finalization, and connection recovery.

Audio sources

Microphone and game-audio paths.

Human speech and game audio use separate capture behavior, buffering, silence thresholds, temporary files, and transcription timing.

Audio quality

Noise and transcript rejection.

Quiet clips, short clips, corrupted input, implausibly long transcripts, and repeated audio events are filtered before they reach the model.

Recognition recovery

Recorder and connection recovery.

Repeated recorder failures trigger recovery attempts. Lost recognition connections reconnect without requiring a full runtime restart.

Audio classification

Non-speech event detection.

Environmental and game sounds can be classified, deduplicated by recency, and inserted into the main and secondary character contexts.

Visual input / media analysis

Frame-based analysis for animated media.

Animated images and videos are downloaded, sampled into representative frames, analyzed as a sequence, and summarized for conversation.

Fast vision

Low-latency visual routing.

When a live response matters more than maximum detail, a faster visual path can describe an image and return control to the response queue quickly.

GIF vision

Animated-image interpretation.

Multiple frames are selected from an animated image so motion, changes, and the visible punchline are represented in the description.

Video vision

Short-video frame sampling.

Video content is converted into a manageable frame set, analyzed for the relevant action, and removed after processing.

Game vision

Game-specific visual context.

Game screens and captured gameplay can be interpreted with prompts that focus on the current game, visible state, action, danger, or objective.

Vision output

Visual descriptions become speech.

Visual analysis can be placed into context, used to generate a reaction, and routed through speech so Moon can respond aloud.

Voice out

Speech-provider failover.

Speech services are prioritized, swapped on failure, streamed when possible, and cleaned up after playback.

Audio routing

Independent output routing.

Voice, music, chat, microphone, local output, and community output can be routed independently.

Interruption

Speech playback state.

Pause, skip, cancel, overlap prevention, speech tokens, and long-playback guards control output.

3D environment

Complete multi-character scene control.

The visual system is a persistent 3D environment with a map, multiple characters, several light sources, cameras, props, clothing, particles, collision, and independently controlled scene objects. The runtime changes this environment continuously during conversation and audience interactions.

Character entities

Independent position and rotation control.

Each 3D character can be positioned, rotated, shown, hidden, moved through the map, framed by cameras, and restored to a known scene position.

Character animation

Independent animation state per character.

Moon, the streamer, secondary characters, mascots, and guests can receive separate animation commands, timing, speed, reset, and idle behavior.

Scene graph control

Map objects and scene state.

The runtime controls characters, cameras, lights, images, overlays, props, effects, and other scene objects as parts of one coordinated 3D graph.

3D command execution

Scene commands from AI and runtime events.

Performance controls resolve to expressions, movement, character transforms, props, cameras, image stages, lighting, particles, and timed transitions. Temporary effects expire without erasing newer state.

Visual state

Animation and expression resolution.

Aliases, exact matches, nearest matches, and fallback selection turn imperfect model output into usable stage control.

Camera

Programmatic 3D camera control.

Cameras support direct transforms, presets, smooth movement, relative framing, random selection, target orbits, shake, saved states, and restoration.

Locks

Uninterruptible presentation state.

Long performances and games protect their timing while ordinary work waits behind them.

Facial capture

52-channel expressive face tracking.

The streamer character uses a high-detail facial capture set that tracks brows, eyes, cheeks, jaw, lips, mouth shapes, tongue, and asymmetric expressions. Moon and other characters use their own face-control paths.

Hand capture

Expressive finger and hand tracking.

Tracked hands include wrist orientation, finger articulation, gesture changes, and live movement. Hand data can also drive attached particle trails.

Body capture

Live body motion input.

Characters can receive tracked body movement or scripted animation. The runtime can switch between capture, animation, and fixed scene control as needed.

Environment map

Persistent navigable 3D set.

The characters occupy a shared map rather than isolated portrait frames. Scene actions can reposition them, move cameras around them, and use locations as part of a sequence.

Lighting system

Multiple controllable light sources.

The environment contains several lights with scene-specific and audio-reactive behavior. Performance commands can alter the lighting state alongside cameras, animation, and effects.

Wardrobe system

Clothing and mesh state control.

Character outfits and forms are assembled from controllable clothing and mesh states. Changes persist in character state and can affect later behavior and presentation.

Prop system

Character-attached and scene props.

Weapons, instruments, food, drinks, phones, generated images, and other objects can be attached, displayed, animated, hidden, and reset on a timed sequence.

Particle system

Tracked and scripted particle effects.

Particles can follow hands, mark attacks, enhance transformations, respond to performances, or run as timed environmental effects.

Character physics

Collision volumes and ragdoll response.

3D characters use collision boxes and physical response. Character-to-character actions can produce contact, displacement, and ragdoll behavior.

Connection

3D engine connection recovery.

Socket initialization, reconnect attempts, listener cleanup, queue clearing, timer cancellation, and stale transform removal handle a lost scene process.

Transport safety

Payload and buffer limits.

Actions and data are size-checked. Excessive socket buffering is detected before it can accumulate unbounded stale scene updates.

Scene inspection

Transform and animation reads.

Camera position, camera rotation, character transforms, and current animation state can be requested and recovered from runtime logs.

Expression timing

Serialized expression queue.

Expressions are queued, deactivated before replacement, held for a controlled duration, and cleared when a neutral face is required.

Animation timing

Serialized animation queue.

Animations run in order, receive duration and metadata handling, trigger an idle reset, and can select a random group member.

Character routing

Secondary-character animation.

Separate animation commands and timed resets can target another character without changing Moon's active animation.

Asset telemetry

Missing asset tracking.

Unknown expressions and animations are recorded, counted, and used to identify gaps between model output and available assets.

Props and scene effects

Timed prop state changes.

Instrument, weapon, confetti, drink, snack, caller image, sleep state, speed changes, and scene-specific effects are activated and reset around animation timing.

Caller imagery

Generated caller presentation.

A caller image can be selected or generated, copied into the avatar scene, displayed, hidden, and paired with an effect.

Avatar state

Sleep and speed controls.

The runtime can set sleep state and animation speed independently of the current speech or interaction.

Consumable props

Timed drink and snack states.

Consumption animations coordinate prop visibility, facial expression, short animation duration, and prop removal.

3D scene control graph

Scene-side animation and effects runtime.

A substantial part of the performance system runs inside the 3D scene. It receives commands, controls character tracking, executes camera programs, manages collision and effects, drives presentation sequences, and returns state to the main runtime.

Motion capture

Per-character face and body tracking.

Moon, the streamer, secondary characters, and the chat mascot can use separate face, body, and hand-tracking paths. Tracking can be enabled or disabled independently.

Lip synchronization

Independent character audio feeds.

Separate audio lines drive lip sync for Moon, the streamer, the on-screen secondary character, and the chat mascot. Simultaneous characters do not share one mouth-control signal.

Mascot animation

Mascot body and speech control.

The chat mascot has its own animation, body-control, face-reset, lip-sync, entrance, effect, and socket-command paths.

Physical simulation

Character collision and ragdolls.

Characters have collision volumes and ragdoll behavior. Mascot movement can physically collide with the streamer or other scene characters.

First-time chatters

Mascot welcome entrance.

When a new chatter appears, the mascot can enter, wave, display the chatter's name, and deliver a welcome message.

Mascot interactions

Physical hug presentation.

An audience interaction can send the mascot toward Moon or the streamer, play the approach and hug animation, resolve contact, and restore the mascot afterward.

Camera control graphs

Preset, relative, random, orbit, and saved cameras.

Scene-side camera programs provide quick positions, relative framing, randomized views, target orbits, cinematic paths, serialized camera saves, and restoration. They are reused by performances, interactions, collabs, and special attacks.

Advanced camera motion

Dynamic scene orbit effects.

Camera controllers can orbit characters or scene objects, vary framing during an action, and produce motion beyond fixed camera presets.

Collaboration layout

Automatic 2D guest positioning.

During a collaboration, a 2D guest can be added to the scene, positioned automatically, and framed with the 3D cast.

Generated-character reveal

3D gacha reveal animation.

The scene runs the visual pull sequence for generated characters, including reveal timing, image staging, camera changes, effects, reaction space, and cleanup.

Cinematic attack

Multi-camera special attack sequence.

An anger-triggered finishing move coordinates music, several camera angles, poses, attack animations, props, effects, timing, and scene restoration as one cinematic.

Horror presentation

Scary poses and sanity effects.

Audience interactions can trigger threatening walks, orbiting cameras, weapon presentation, unsettling visual effects, and timed horror sequences.

Reactive lighting

Audio-driven scene lights.

Scene lighting can react to selected audio sources, allowing music, voices, and performance audio to drive intensity or visual response.

Hand effects

Tracked particle trails.

Hand tracking can emit particle trails that follow live hand movement during gestures, dances, attacks, and performance sequences.

Rendering optimization

Low-cost character shadows.

A custom shadow representation reduces rendering cost while preserving a readable character shadow in the scene.

Community streaks

Streak-triggered animations.

Live-platform streak events can dispatch scene animations and effects that acknowledge sustained community activity.

Scene command routing

Mascot socket control.

The mascot receives dedicated socket commands for animation, face, movement, entrances, reactions, and other audience-driven scene actions.

Food gifts

Audience-triggered food deliveries.

Chat can send cookies, cake, pizza, drinks, and snacks into the scene through dedicated prop and presentation sequences.

Retaliation system

Targeted attacks and counterattacks.

Chat can attack Moon or the streamer. Attacks against Moon can select from several counterattack sequences with their own animations, effects, and timing.

Character abilities

Secondary-character spell sequences.

On-screen characters can execute their own spell and ability presentations with character-specific animation, effects, audio, and scene commands.

Legacy mascot support

Older mascot scene behavior.

The scene retains support for a previous chat mascot and its dedicated spell presentation, preserving older audience interactions.

Stream introduction

Intro video playback.

A dedicated intro sequence starts the stream presentation, plays the configured video, and coordinates the scene transition into normal operation.

Interactive experiences / staged execution

Channel-point interaction lifecycle.

Small audience actions can trigger a quick reaction. Larger ones prepare generated media, stage a sequence, update overlays, publish results, and restore normal operation.

Fast reactions

Immediate audiovisual effects.

Affection, chaos, outfits, sound cues, physical comedy, opinions, attention, and character reactions execute with minimal preparation.

Generated media

Generation, analysis, and presentation.

Physical reactions

Targeted character interactions.

Audience actions can target Moon, another character, a guest, or a shared scene and produce different audio, expression, movement, and response behavior.

Forms and costumes

Persistent outfit state.

Form changes alter the visual state, affect context reminders, trigger presentation behavior, and can be reverted through a separate reset path.

Conversation actions

Prompted audience responses.

Requests for attention, opinions, ratings, roasts, hot takes, answers, memories, and user-message highlights enter the conversational context.

Sound actions

Effects and audio cues.

Short effects, stingers, impacts, environmental sounds, music changes, applause, and rejection cues can run alongside speech and movement.

Performance actions

Dance, speed, and song control.

Longer performances coordinate music, animation speed, camera state, speech, timing, and a protected return to the normal scene.

Full-song production

Time-coded music and 3D scene performance.

A full song can run as a protected production with separate music and vocal channels, a timed performance script, dynamic camera moves, character transforms, facial expressions, animations, dance actions, speed changes, scene commands, and post-song restoration.

Performance script

Timestamped cue dispatch.

Song cue files can schedule lyrics, expressions, animations, camera moves, character-specific actions, speed changes, model context updates, and arbitrary scene commands at precise offsets.

Camera console

Dynamic camera and transform control.

Performance cues can move the camera smoothly, change the character transform, use close or wide views, and restore the talk position when the song ends.

Vocal presentation

Vocal channel and lip-sync coordination.

Music and vocals are routed separately so the performance can drive the voice path, character mouth movement, and community or local audio outputs independently.

Scene effects

Animation, dance, and action cues.

Timed cues can select animations, shuffle expressions, change animation speed, trigger character actions, activate props, and dispatch configured scene or lighting actions.

Performance recovery

Protected start and clean return.

The system stops ambient pulses, freezes conflicting animation changes, saves the camera state, handles errors, resets expressions and animation, restores the camera, and releases the performance lock.

Media reports

Review and broadcast formats.

Generated or retrieved content can be presented as a review, report, advertisement, phone call, comic, stylized image, or informational segment.

Utility actions

Daily and community state.

Greetings, snacks, drinks, cookies, attention, user records, answer enforcement, and small status actions connect routine chat to persistent state.

Interaction registry

Modular interaction discovery.

Interactive scripts are discovered, normalized into user-facing labels, loaded independently, and isolated so one failed module does not disable the complete set.

Execution model

Preparation and execution phases.

Expensive image, vision, model, and speech work can prepare in the background. The visible sequence begins later when the scheduler grants execution.

Execution model

Immediate and queued paths.

Short effects can run immediately. Longer interactions enter the priority queue and wait behind protected work.

Restart behavior

Pending interaction recovery.

Serializable labels, users, messages, and prepared data can be restored after a reconnect. Live functions and promises cannot.

Audience output

Multi-surface result publishing.

Interactions can update the avatar, local audio, community audio, chat, overlays, images, timers, leaderboards, and persistent history.

Access and limits

Selected interactions enforce user access, cooldowns, image-generation limits, active-game rules, and current scene restrictions.

Collection systems

Randomized character and item draws.

Character, hero, product, voice, image, and outcome selection is combined with quotas, staged presentation, reaction speech, and cleanup.

Audience games

Timed quiz and leaderboard state.

Questions, answer boards, guesses, semantic matching, scores, timers, leaderboards, celebration, and a closing response form one loop.

Media

Timed media playback.

Music, clips, effects, generated speech, applause, and transitions are coordinated with the visual scene.

Publishing

Audience-facing result output.

Images, outcomes, messages, timers, and announcements can be shown to the audience in several places.

Staged content productions

Review, report, advertisement, and call sequences.

Content generation can be followed by visual inspection, scripted presentation, avatar reaction, speech, overlay updates, community publishing, and camera restoration.

Presentation timing

Minimum-duration and queue-aware staging.

Generated content can remain visible for a required duration, shorten under queue pressure, or defer a reaction until the scene is available.

Interaction cleanup

Temporary-file and scene restoration.

Generated files, temporary images, audio, overlay states, camera changes, props, and presentation locks are cleared after an experience.

System boundary

AI runtime plus 3D production environment.

The system manages event timing, context selection, response validation, media preparation, multi-character scene control, cameras, lighting, tracking, physics, interruption, recovery, and persistence as one runtime.

Screen characters

Operator surface

Phone characters

Contextual phone-call conversations.

A caller can reach a selected lore character through a staged phone sequence. The system manages caller identity, phone UI, ring and dial audio, character-specific dialogue, short-turn constraints, memory updates, live subtitles, and hang-up restoration.

Phone state

Call lifecycle and interruption rules.

Active calls reserve the required presentation state, prevent conflicting phone events, route both sides of the conversation into context, and restore the scene when the call ends.

Chat mascot

Dedicated mascot persona.

A separate chat-facing mascot can speak with its own voice, expressions, movement, effects, and audience-triggered actions while Moonlash remains the primary host.

Stream operations

Timed advertising and break control.

Ad breaks have countdown timing, start and end cues, music, overlay updates, queue interaction rules, and return-to-stream behavior.

Operator input

Keyboard and hardware hotkeys.

Local key events can trigger stream actions, media, camera changes, avatar controls, emergency stops, and other operator functions without using chat.

Local command transport

Persistent socket command channel.

External overlays and control surfaces can send and receive lightweight event messages for scene changes, timers, media, status, and operator actions.

Control API

HTTP control endpoints.

Local tools and hardware integrations can invoke runtime actions through a small request-based control layer, including avatar, stream, queue, and media operations.

Live platform protocol

In-house event and API client.

The live-platform connection was implemented directly in the project. It handles authentication checks, persistent event sessions, subscription registration, reconnects, duplicate suppression, event normalization, chat, follows, raids, cheers, and channel-point events.

Discord

Persistent 24/7 Discord presence.

Moon remains available in Discord outside stream hours. Discord uses the same core persona, memory, model routing, response validation, vision, image generation, speech, secondary characters, and history systems as the live runtime.

Discord text

Channels, direct messages, mentions, and replies.

Moon routes selected channels, direct messages, explicit mentions, replies to earlier messages, stickers, and referenced-message context into the conversation system.

Discord context

Channel history and identity resolution.

Recent channel messages, reply chains, channel names, mentions, aliases, and resolved user identities provide context before Moon responds.

Discord media

Image, animated image, and video vision.

Attachments and linked media are downloaded, converted, sampled into frames when needed, analyzed, added to context, and removed after processing.

Discord links

Embed and shared-media interpretation.

Video embeds, thumbnails, titles, authors, refreshed attachment links, and supported media-sharing links are converted into useful conversational context.

Discord generation

Image generation and publishing.

Users can request several image formats and character styles. Requests use prompt checks, cooldowns, quotas, loading messages, visual analysis, context updates, and result publishing.

Discord memory

Memory, history, and character commands.

Discord commands can load user memories, add long-term notes, search history, list characters, inspect prompts, check quotas, and request specialized visual output.

Discord voice

Continuous voice-channel conversation.

Moon can join a voice channel, receive compressed user audio, decode and measure it, reject noise, transcribe completed turns, identify speakers, add speech to context, generate a response, and play her voice back into the channel.

Discord voice input

Speaker-aware recording and transcription.

Speaking events open per-user recordings. Inactivity closes each turn, volume and duration checks reject bad clips, and transcript anomaly checks prevent false input.

Discord voice output

Converted and retried audio playback.

Generated speech is converted to the required voice format, retried after conversion failure, volume-controlled, played through the shared channel connection, and cleaned up.

Discord characters

Character-specific voice routing.

Mentions and conversation state can route speech to Moon or a screen character. Each character keeps separate context, voice, animation, expression, and lip-sync behavior.

Discord operating modes

Stream, watch-along, and offline behavior.

Channel selection and response rules change when the stream is live, when Discord is the active broadcast environment, during watch-alongs, or while Moon is available offline.

Integrated runtime

Asynchronous subsystem coordination.

Chat, voice, vision, games, timers, controls, model calls, media preparation, character state, and scene commands are scheduled through one runtime with explicit priority and interruption rules.

Conversational intelligence

The conversational layer maintains long-running context, user memory, scenario state, model routing, validation, repair, summarization, and structured history.

Multimodal perception

3D production environment
